<h2 id="about"> 🔹 О проекте</h2>

<p align="justify">
  <strong>«Транскрибация Встреч»</strong> — это специализированное веб-приложение, предназначенное для автоматического протоколирования рабочих, учебных и деловых совещаний на основе аудио- и видеозаписей. Продукт ориентирован на полную конфиденциальность данных и автономную работу в закрытом сетевом контуре организации (On-Premise), без отправки информации на внешние облачные серверы.
</p>

### Решаемые «боли» целевой аудитории (Team Leads, Product/Project Managers):
*   **Экономия времени:** автоматическая расшифровка и разметка реплик освобождают от необходимости тратить 4+ часов в неделю на ручное составление протоколов и переслушивание многочасовых записей встреч.
*   **Сохранение договорённостей:** предотвращение потери важных решений и требований, принятых в процессе длительного или многословного обсуждения.
*   **Исключение споров:** фиксация конкретных исполнителей и дедлайнов по таймкодам, что повышает общую продуктивность команды в среднем на 25%.

<h2 id="functions"> 🔹 Функциональные возможности</h2>

<ul>
  <li><strong>Авторизованный доступ</strong> — защищённая регистрация и вход в личный кабинет с верификацией пароля.</li>
  <li><strong>Личный кабинет пользователя</strong> — хранение персонального списка встреч с отображением метаданных (дата, длительность, количество участников).</li>
  <li><strong>Удобный загрузчик медиа</strong> — поддержка популярных форматов (MP3, WAV, MP4, AVI, MOV) с валидацией размера и заполнением названия/описания встречи.</li>
  <li><strong>Интерактивная расшифровка (Smart Player)</strong> — синхронизация аудиоплеера с текстом. Клик по таймкоду реплики автоматически перематывает запись на соответствующий момент.</li>
  <li><strong>Фильтрация по спикерам</strong> — возможность скрывать реплики других участников, чтобы сосредоточиться на вкладе конкретного человека.</li>
  <li><strong>Ручное редактирование</strong> — изменение имён спикеров (замена системных меток <code>SPEAKER_01</code> на реальные имена участников).</li>
  <li><strong>Административная панель</strong> — мониторинг сводных показателей (всего пользователей, встреч, общая длительность, занятое дисковое пространство), управление таблицей пользователей и ручное создание учётных записей.</li>
</ul>

<h2 id="architecture"> 🔹 Архитектура и бизнес-процессы (BPMN)</h2>

<p align="justify">
  Бизнес-процессы системы спроектированы в стандарте BPMN и разделены на верхнеуровневый процесс и детализированные подпроцессы для обеспечения стабильной обработки и удобной навигации по результатам.
</p>

### 1. Верхнеуровневый процесс работы системы (BPMN)
<p align="justify">
  После авторизации пользователь выбирает медиафайл. Система выполняет валидацию формата и размера. В случае ошибки формируется уведомление, и процесс циклически возвращается к выбору файла. При успешной валидации управление передаётся подпроцессу <strong>«Обработка файла»</strong>. По завершении вычислений результат сохраняется в базу данных, статус встречи обновляется, а пользователю отправляется сигнал о готовности.
</p>

### 2. Подпроцесс «Обработка файла»
<ol>
  <li><strong>Подготовка аудиоматериала</strong> — конвертация исходного файла в моно-формат WAV (16 кГц) и нормализация звука средствами <code>FFmpeg</code> для стабильной работы последующих алгоритмов.</li>
  <li><strong>Запуск распознавания речи (ASR)</strong> — преобразование подготовленного звукового сигнала в текстовый поток (без привязки к голосам).</li>
  <li><strong>Выполнение диаризации по спикерам</strong> — анализ акустических характеристик голосов для разграничения участников совещания.</li>
  <li><strong>Формирование таймкодов реплик</strong> — синхронизация полученного текста и данных о спикерах с временными метками оригинального файла.</li>
</ol>

### 3. Подпроцесс «Просмотр результата»
<p align="justify">
  После загрузки метаданных встречи и отображения структурированной расшифровки пользователю доступны три сценария работы: фильтрация контента по конкретным спикерам, навигация по записи с помощью кликов по таймкодам реплик, либо комбинирование этих сценариев для быстрого поиска нужного фрагмента.
</p>

<h2 id="metrics"> 🔹 Критерии работоспособности и метрики</h2>

### 1. Функциональные метрики качества
*   **Качество распознавания (ASR):** WER (Word Error Rate) $\le$ 20% на наборе тестовых русскоязычных записей.
*   **Точность разделения голосов (Diarization):** DER (Diarization Error Rate) $\le$ 25%. Поддержка $\ge$ 2 спикеров в одном аудиофайле.
*   **Точность таймкодов:** погрешность сопоставления временных меток к фрагментам текста составляет $\le \pm0.5$ сек относительно исходного аудио.
*   **Выходной формат:** структурированный JSON со следующими обязательными полями: `speaker_id`, `start_time`, `text`.

### 2. Технические ограничения и производительность
*   **Локальность:** 100% офлайн-инференс. Полное отсутствие исходящих сетевых запросов к внешним API/облакам в процессе работы.
*   **Скорость обработки:** 1 минута аудио обрабатывается $\le$ 40 сек на CPU и $\le$ 15 сек на GPU.
*   **Стабильность:** непрерывная обработка файлов длительностью до 60 минут без аварийных завершений и утечек оперативной памяти.
*   **Устойчивость:** корректная работа алгоритмов при уровне сигнал/шум SNR $\ge$ 15 дБ и перекрытии речи разных спикеров (overlap) $\le$ 15% от общей длительности.

### Анализ косвенных конкурентов:
*   *Otter.ai, Descript:* облачные зарубежные SaaS-решения. Демонстрируют высокую точность, однако требуют обязательной передачи аудио на сторонние зарубежные серверы, что неприемлемо по соображениям корпоративной безопасности.

<h2 id="setup"> 🔹 Установка и запуск</h2>

<h3>Инструкция локального развертывания</h3>

<ol>
  <li><strong>Клонирование репозитория:</strong>

```bash
git clone git@github.com:MrCrise/Verbatim.git
cd Verbatim
```
    
  </li>

  <li><strong>Создание конфигурационного файла среды:</strong>
    <p>Создайте в корневом каталоге файл <code>.env</code> со следующим содержимым:</p>

```bash
# --- DATABASE ---
POSTGRES_SERVER=postgres
POSTGRES_USER=postgres
POSTGRES_PASSWORD=password
POSTGRES_DB=verbatim_db
POSTGRES_PORT=5432

# --- JWT ---
JWT_SECRET_KEY=  # Сюда необходимо вставить ваш секретный JWT ключ.

# --- REDIS ---
REDIS_HOST=redis
REDIS_PORT=6379

# --- MINIO ---
MINIO_ENDPOINT=localhost:9000
MINIO_ACCESS_KEY=admin
MINIO_SECRET_KEY=password123  # Сюда необходимо вставить ваш пароль от объектного хранилища.
MINIO_BUCKET_AUDIO=raw-audio
MINIO_KMS_SECRET_KEY=""  # Сюда необходимо вставить ваш секретный KMS ключ.

# --- ML ---
HF_TOKEN=  # Сюда необходимо вставить ваш HuggingFace токен.
```

   > **Важно:** Перед запуском вы должны зайти на HuggingFace под своим аккаунтом и вручную принять пользовательские соглашения для моделей [pyannote/speaker-diarization-3.1](https://huggingface.co/pyannote/speaker-diarization-3.1) и [pyannote/segmentation-3.0](https://huggingface.co/pyannote/segmentation-3.0).

  </li>

  <li><strong>Запуск всей инфраструктуры:</strong>

  ```bash
  docker-compose up -d --build
  ```

  *Примечание: Первая сборка проекта может занять 10-15 минут, так как происходит скачивание образов CUDA, PyTorch и весов модели GigaAM.*

  </li>

  <li><strong>Инициализация базы данных и применение миграций:</strong>

  ```bash
  docker-compose exec backend alembic upgrade head
  ```

  </li>

  <li><strong>Настройка Объектного Хранилища (S3)</strong></li>

  1. Откройте панель управления MinIO в браузере: `http://localhost:9001` (Логин: `admin`, Пароль: `password123` или тот, который вы указали в `.env`).
  2. Перейдите в раздел **Buckets** -> **Create Bucket**.
  3. Создайте бакет с именем **`raw-audio`**. 
   *(MinIO автоматически настроит локальное шифрование AES-256 благодаря ключу KMS, указанному в `.env`).*

</ol>
